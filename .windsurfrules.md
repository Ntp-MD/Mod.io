# Always read variables.css before writing CSS

Cleaner code - No redundant class definitions
Consistent styling - Single source of truth for each class

# Core CSS Rules

- **No Auto Create Variables**: Use only existing CSS variables from variables.css
- **No Hard-coded Values**: Use variables from existing varaibles.css only
- **Variables**: Use only existing CSS variables
- **Layout**: Use flexbox/grid, avoid float. Use gap for spacing, margin for exceptions
- **Units**: Use px for fixed values, clamp() for responsive typography
- **Calc()**: Prefer *1.5, *2, *3 multiplication. Decimal (*1.5, \*2.5) only when necessary. Division /2, /3, /4 allowed
- **Naming**: BEM convention (block-element-modifier), max 3 levels. Use separate classes like "btn primary-color"
- **Performance**: Class-based selectors only, minimize reflows, use CSS containment
- **Validation**: Check for hard-coded values before committing

## Grid-Mod System (grid-mod.js)

### **Core Usage**

- **Attribute Format**: `grid="large,medium,semi,small,gap"`
- **Examples**:
  - `grid="6,4,3,1,var(--gap-md)"` - Full responsive
  - `grid="1,var(--gap-md)"` - Single column with gap
- **Individual Attributes**: `large`, `medium`, `semi`, `small`, `gap` (fallback)

### **Breakpoint System**

- **Small**: ≤480px (mobile)
- **Semi**: ≤992px (tablet)
- **Medium**: ≤1200px (desktop)
- **Large**: >1200px (wide)

### **Configuration**

- **TogglePrecision**: false = simple grid template, true = complex auto-fit
- **Performance**: Throttled resize events (16ms), caching, batch DOM operations
- **Auto-Initialization**: DOM ready, mutation observer, intersection observer

### **Usage Guidelines**

- **When to Use**: Components with 3+ items requiring complex responsive behavior
- **When NOT to Use**: Simple 2-column layouts, fixed layouts (Hero, CTA, Header, Footer)
- **Local Variables Exception**: Grid-mod classes can use local variables for advanced responsive behavior

### **Performance Features**

- **Smart Observer**: Single global mutation observer
- **Lazy Initialization**: Intersection observer for off-screen elements
- **Throttling**: 60fps resize handling
- **Memory Management**: Cleanup functions for observers and caches

## Example Usage

```css
.good-component {
  color: var(--main-color-2);
  background: var(--color-primary);
  padding: var(--gap-md);
  margin: 0 var(--gap-sm);
  font-size: var(--font-md);
  border-radius: var(--radius-md);
  transition: var(--transition-fast);
}
```

- **Use Gap on Parent**: For flex/grid containers with multiple children that need consistent spacing, use `gap` on the parent container instead of `margin-bottom` on individual children.
- **When to Use Gap**: Flex/Grid containers with 3+ elements, consistent spacing across all children, responsive layouts, cleaner code maintenance.
- **When to Use Margin**:
  - **Single element spacing**: Individual elements that need specific spacing without siblings
  - **Asymmetric spacing**: Different spacing needed for top vs bottom (top ≠ bottom)
  - **Override default spacing**: When you need to override parent gap spacing
  - **Negative spacing**: For special positioning effects (negative margins)
  - **Responsive exceptions**: Different spacing rules for specific breakpoints
  - **Complex positioning**: Advanced layout requirements that gap cannot handle

- **Individual Elements Exception**:
  - **Single elements without siblings** may use hard-coded values (px, %, vh, etc.)
  - **Specific design requirements** like hamburger menu lines, icon parts, borders
  - **Special positioning** where no suitable variable exists
  - **Performance critical** elements where direct values are optimal
- **Modern Approach**: Prefer `gap` for modern CSS Grid/Flexbox layouts. Only use margins for exceptional spacing needs.

- **Advanced Gap Usage**:
  - Use gap with CSS variables: `gap: var(--gap-name)`
  - Use gap in calc() for dynamic spacing: `gap: calc(var(--gap-name) * 1.5)`

  - **Calc() Flexibility**:
  - **Preferred multiplication**: *1.5, *2, \*3 (non-decimal)
  - **Decimal multiplication**: \*1.5, \*2.5 (use only when necessary)
  - Division allowed: /2, /3, /4

## Grid-Mod Usage Guidelines (grid-mod.js)

- **When to Use Grid-Mod**: Use for components with 3+ items that require complex responsive behavior (e.g., Features, HowItWorks, Testimonials, Pricing).
- **When NOT to Use Grid-Mod**: Avoid for simple 2-column layouts or fixed layouts (e.g., Hero, CTA, Header, Footer). Use regular CSS grid or flexbox instead.
- **Grid-Mod Attributes**: Use `large`, `medium`, `semi`, `small` attributes for responsive breakpoints, and `gap` attribute for spacing.
- **Local Variables Exception**: Grid-mod classes can use local variables for advanced responsive grid behavior (this is the only exception to the "no new variables" rule).

### **Variable Violation Resolution Process**

#### **Rule: No Auto-Create Variables**

- **Never create new CSS variables** without explicit approval
- **Always use existing variables** from variables.css
- **Replace missing variables** with closest equivalent

#### **Common Missing Variables & Solutions:**

**1. `--color-border`**

- **Replace with**: `--main-color-8` (#e5e5e5)
- **Usage**: `border: 1px solid var(--main-color-8)`

**2. `--color-background`**

- **Replace with**: `--color-white` (#ffffff)
- **Usage**: `background-color: var(--color-white)`

**3. `--color-text`**

- **Replace with**: `--main-color-2` (#171717)
- **Usage**: `color: var(--main-color-2)`

**4. `--color-text-muted`**

- **Replace with**: `--main-color-6` (#737373)
- **Usage**: `color: var(--main-color-6)`

#### **Resolution Steps:**

1. **Identify missing variables** using validation script
2. **Find closest equivalent** from existing variables
3. **Replace all occurrences** across CSS files
4. **Test functionality** to ensure visual consistency
5. **Update documentation** with variable mapping
